\Autoref{fig:eo-syntax} defines syntax for the
fragment of Eunoia considered in this document, where the
metavariable $s$ ranges over a set of \emph{symbols}.
%
First, we define grammars for the core term syntax.
%
Each term is either a symbol $s$, an (integer/rational/string)
\emph{literal} $Ôäô$, an \emph{application} $\eapp{s}{\vec t}$,
or a \emph{binder application} $\eapp{s}{\epar{\vec ¤ü}\ t}$.
%
Each \emph{parameter} $¤ü$ consists of a symbol $s$, a term
% forward ref to where we discuss typing ??
$t$ (the type of the parameter), and possibly a
\emph{parameter attribute} $╬¢$.
%

\subsection{Built-in Operators.}
%
Eunoia has a fixed set of \emph{built-in symbols}
(\autoref{fig:eo-syntax}) that allow manipulating the
syntactic structure of terms.
%
In particular, Ethos will eagerly evaluate
applications of some built-in symbols to perform efficient
computations on (integer, rational, string, bit-vector)
literals during proof-checking.
%%

A distinctive feature of Eunoia is its treatment of
$n$-ary operators (see \autoref{sec:eo-elab}):
given a right-associative operator $f$ with nil terminator
$t_\nil$, the \emph{$f$-list} with elements $\plur t n$
is the nested application $\eapp{f}{t_1\ \eapp{f}{\ldots\ \eapp{f}{t_n\ t_\nil}}}$.
%
The built-in symbols for lists operate generically on
these term structures.

Eunoia also provides the heterogeneous list type $\eo{List}$
(with constructors $\eo{List::cons}$ and $\eo{List::nil}$),
and the $\eo{var}$ operator (which constructs a `variable'
from a string literal and a type).
%
Together these operators are used to encode variable
lists in quantifiers and binders.

\subsection{Signatures and Proof Scripts.}\label{sec:eo-sig}

The commands of Eunoia are divided into
\emph{standard commands} and \emph{proof commands}.
%
Hereinafter, a Eunoia \emph{signature} is a list $╬ö$
of standard commands, and a \emph{proof script} is a
list $╬Ñ = \prn{\vec ╬┤ Ô¿¥ \vec ¤Ç}$
where $\vec ╬┤$ is a list of standard commands called
the \emph{preamble} and $\vec ¤Ç$ is a list of
proof commands called the \emph{body} of the script.
%
In practice, a Eunoia-friendly SMT solver provides a
signature $╬ö$ encoding the inference rules that may
be used by its proof scripts.
%
Then, the correctness of a proof script $╬Ñ$ may then be
checked wrt. $╬ö$ via Ethos.

In the following discussion, we define some terminology
and notation for commands, signatures, and proof scripts.
The resulting interface is useful for
characterizing the elaboration and translation operators
in \autoref{sec:eo-elab} and \autoref{sec:results}.

\subsubsection{Constant Declaration.}
%
Let $╬┤$ be a \emph{(parameterized) constant declaration}
with symbol $s$, parameters $\vec ¤ü$, and term $t$.
%
We may write $\prn{╬┤ Ôèó s(\vec ¤ü) : t}$ to express
that the command $╬┤$ declares $t$ as the \emph{type} of $s$
with respect to parameters $\vec ¤ü$.
%
Furthermore, if a constant attribute $╬▒$ is given by $╬┤$,
we may write $\prn{╬┤ Ôèó s(\vec ¤ü) ÔêÀ ╬▒}$.
%
\subsubsection{Macro Definition.}
%
Let $╬┤$ be a \emph{macro definition} with
symbol $s$, parameters $\vec ¤ü$, and term $t$.
%
Then $╬┤$ declares $t$ as the \emph{definiens} of $s$
wrt. $\vec ¤ü$; written $\prn{╬┤ Ôèó s(\vec ¤ü) Ôëö t}$.
%
If the attribute $\attr{type}\ t'$ is given by $╬┤$,
then $\prn{╬┤ Ôèó s(\vec ¤ü) : t'}$ also holds.
%
\subsubsection{Program Declaration.}
%

symbol $s$, parameters $\vec ¤ü$,
$\attr{signature}\,\prn{\plur t n}\,t'$,
and cases $\plur c m$.
%

where $\plur t n$ are the \emph{domain} types of $s$
and $t'$ is the \emph{range}:

	\quad\text{and}\quad
	╬┤ Ôèó s(\vec ¤ü) Ôëö \mbf{cases}[c_1, \ldots, c_m]

\subsubsection{Proof Commands.}
Each $¤Ç$ is either an \emph{assumption}
(i.e., $\epar{\kw{assume}\ s\ ¤å}$ or
$\epar{\kw{assume-push}\ s\ ¤å}$)
or a \emph{step}
(i.e.,
$\epar{\kw{step}\ s\ ¤å\ \ldots}$ or
$\epar{\kw{step-pop}\ s\ ¤å\ \ldots}$).
%
In any case, we call
$s$ the \emph{name} of $¤Ç$ and $¤å$ the \emph{conclusion},
written:
$$¤Ç Ôèó s : \eapp{\mtt{Proof}}{¤å}$$

When $¤Ç$ is a step with
$\attr{rule}\ r$,
the definiens of $s$ is the result of
applying any terms given by
$\attr{args}\ \epar{\plur t m}$,
and
$\attr{premises}\ \epar{\plur p n}$, e.g.;
%
$$
	¤Ç Ôèó s Ôëö \eapp{r}{\plur t m\ \plur p n}
$$

\subsection{Elaboration of Terms}\label{sec:eo-elab}
%
% TODO.
% - need to define 'normal form' of Eunoia terms: the head
%   of every application must either be a builtin or a program.
Terms are \emph{elaborated} to eliminate all $n$-ary
applications and distinguish between constant-level and
program-level function applications.
%
We say that a term $t$ is in \emph{normal form} wrt. a
signature $╬ö$ when the head symbol of every application in
$t$ is either a built-in or a $╬ö$-program.

Consequently, an application $\eapp{f}{t}$ of an
unattributed constant symbol $f$ has the normal form
$\eapp{\mtt{\_}}{f\ t}$, where $\mtt{\_}$ is the Eunoia
built-in for higher-order application.
%
In general, the normal form of $\eapp{f}{\vec t}$
is determined by the attribute assigned to $f$ by $╬ö$.
%
% %
% For example, consider a constant declaration $╬┤$ with:
% $$
% 	╬┤ Ôèó \eor : \earr{\bool\ \bool\ \bool}
% 	\quad\text{and}\quad
% 	╬┤ Ôèó \eor ÔêÀ {\rcn \false}
% $$
% % and $╬┤ Ôèó $.
% % $$\dca{\eor}{\earr{\bool\ \bool\ \bool}}$$
% %
% From the type of $\eor$, we may expect its only valid
% uses to be of the form $\eapp{\eor}{t_1\ t_2}$.
% %
% However, the assignment of the constant attribute means
% that the $\mtt{or}$ symbol is treated as
% \emph{right-associative} with \emph{nil-terminator} $\false$.
% %
% Thus, $n$-ary applications of $\eor$ are
% \emph{elaborated} to a normal form, e.g.;
% %
% $$\eapp{\eor}{\ev x\ \ev y\ \ev z}\ Ôçó\
% 	\eapp{\eor}{\ev x\ \eapp{\eor}{\ev y\ \eapp{\eor}{\ev z\ \false}}}$$
% %
% Furthermore, the elaboration of such applications may
% also depend on the attributes of `locally bound' symbols.
% For example, consider $\vec ¤ü$ containing
% parameters $\epar{\ev x\ \bool}$,
% $\epar{\ev y\ \bool\ \mtt{:list}}$,
% and $\epar{\ev z\ \bool}$.
% %
% Then, observe:
% $$
% 	\eapp{\eor}{\ev x\ \ev y\ \ev z}
% 	\ {Ôçó}
% 	\ \econs\eor{\ev x}{
% 		\econcat{\eor}{\ev y}{\eapp{\eor}{\ev z\ \false}}
% 	}
% $$
% %
% Here, $\eo{list\_concat}$ is a built-in ternary operator:
% $\eapp{\eo{list\_concat}}{f\ \ell\ \ell'}$ concatenates
% two $f$-lists $\ell$ and $\ell'$.
% %
% The $\mtt{:list}$ attribute alters the elaboration strategy
% under the assumption that $\ev y$ will (eventually) be
% substituted for some $\eor$-list.
% %
% It may be helpful for the reader
% to consider the result of substituting
% $\ev y Ôåª \econs\eor{\ev w}{\false}$
% thus:
% $$
% 	\begin{array}[t]{c}
% 		\econs\eor{\ev x}{
% 			\econcat{\eor}{\econs\eor{\ev w}{\false}}
% 			{\eapp{\eor}{\ev z\ \false}}}
% 		\\[1mm] {Ôçú} \\[1mm]
% 		\econs\eor{\ev x}{
% 			\econs\eor{\ev w}{
% 				\econs\eor{\ev z}{\false}}
% 		}
% 	\end{array}
% $$
% %
We define the $\mbf{elab}$ operator below with the aim
of supporting the elaboration strategies
corresponding to the constant attributes of
\autoref{fig:eo-syntax}.
%
\newcommand{\body}{\text{body}}
\newcommand{\vars}{\text{vars}}
\begin{definition}
	For any $╬ö$ and $\vec ¤ü$, let $\elab{╬ö,\vec ¤ü}$
	be the least operator such that:
	\begin{enumerate}
		\item For any symbol $s$, $\elab{╬ö,\vec ¤ü}[s] = s$.
		      %
		\item For any binder application
		      $\eapp{s}{\epar{\vec v}\ t}$
		      where
		      $╬ö Ôèó s ÔêÀ \bd t_\cons$,
		      $$ \elab{╬ö,\vec ¤ü}[\eapp{s}{\epar{\vec v}\ t}]
			      =
			      \eapp{s}{t_\vars\ t_\body}
		      $$
		      where $t_\vars$ encodes
		      $\vec v = \epar{x_1\ t_1}\ldots\epar{x_n\ t_n}$
		      as a $t_\cons$-list of $\eo{var}$ terms,
		      and $t_\body$ is the result of substituting each
		      $x_i$ in $t$ for its $\eo{var}$ counterpart:
		      %
		      $$
			      \begin{array}{l@{\ {Ôëö}\ }l}
				      t_\vars &
				      \elab{╬ö, \vec ¤ü}[\eapp{t_\cons}{\plur X n}]
				      \\
				      t_\body &
				      \elab{╬ö, \vec ¤ü}
				      [\,
				      \mbf{subst}_{\prn{\vec x, \vec X}}[t]
				      \,]
			      \end{array}
			      \qquad
			      {\small\text{where}\
				      X_i Ôëö \epar{\eo{var}\ \estr{x_i}\ t_i}}
		      $$

		      %
		\item For any application $\eapp s {\vec t}$,
		      let $t_i' Ôëö \elab{╬ö, \vec ¤ü}[t_i]$ in:
		      \begin{enumerate}
			      \item If $s$ is a built-in or $s Ôêê ╬ö.\msf{prog}$, then
			            $\elab{╬ö, \vec ¤ü}[\eapp{s}{\vec t}] =
				            \eapp{s}{\vec t'}$.
			      \item If $s Ôêê \vec ¤ü$, then
			            $\elab{╬ö, \vec ¤ü}[\eapp{s}{\vec t}] =
				            \eapp{\mtt{\_}}{\ldots
				            \eapp{\mtt{\_}}{s\ {t_1'}}\ \ldots\ {t_n'}
				            }
			            $.

			      \item If $╬ö Ôèó s ÔêÀ ╬▒$ for some associative
			            attribute $╬▒$, then:
			            $$
				            \elab{╬ö,\vec ¤ü}[\eapp{s}{\vec t}]\ {=}\
				            \begin{scases}
					            \foldr(G, t_\nil, \vec t')
					            & \tsm{$\prn{╬▒ = \rcn{t_\nil}}$}
					            \\
					            \foldl(G, t_\nil, \vec t')
					            & \tsm{$\prn{╬▒ = \lcn{t_\nil}}$}
					            \\
					            \foldr(G, t'_n, t'_1\ldots t'_{n-1})
					            & \tsm{$\prn{╬▒ = \rc}$}
					            \\
					            \foldl(G, t'_1, t'_2\ldots t'_{n})
					            & \tsm{$\prn{╬▒ = \lc}$}
					            % \\
					            % \elab{╬ö,\vec ¤ü}[\eapp{{t_\cons}}{\mbf{chain}(s, \vec t')}]
					            %  &
					            % \tsm{if $╬ö Ôèó s ÔêÀ \ch {t_\cons}$,}
					            % \\
					            % \elab{╬ö,\vec ¤ü}[\eapp{{t_\cons}}{\mbf{pairs}(s, \vec t')}]
					            %  &
					            % \tsm{if $╬ö Ôèó s ÔêÀ \pw {t_\cons}$,}
					            % \\
					            % \elab{╬ö,\vec ¤ü}[\eapp{{t_\cons}}{\vec t'}]
					            %  &
					            % \tsm{if $╬ö Ôèó s ÔêÀ \al {t_\cons}$,}
					            % \\[1mm]
					            %
					            %  &
					            % \tsm{otherwise.}
				            \end{scases}
			            $$
			            where\footnote{%
				            $\foldl(G, a, l)$ and $\foldr(G, a, l)$ are the
				            standard left and right folds using combining
				            function $G$ and accumulator $a$ on list $l$.
			            }%
			            $G(x,y) Ôëö \glue{\vec ¤ü, s}[x,y]$ is defined as
			            follows:
			            $$
				            \begin{array}[t]{r@{\ Ôëö\ }l}
					            \glue{\vec ¤ü, s}[x,y]
					             &
					            \begin{scases}
						            \econcat{s}{x}{y} & \tsm{$\prn{\vec ¤ü Ôèó x ÔêÀ \attr{list}}$}
						            \\
						            \eapp{\mtt{\_}}{\eapp{\mtt{\_}}{s\ x}\ y}
						            & \tsm{otherwise.}
					            \end{scases}
				            \end{array}
			            $$

			      \item If $╬ö Ôèó s ÔêÀ ╬▒$ for some non-associative
			            attribute $╬▒$, then:
			            $$
				            \elab{╬ö,\vec ¤ü}[\eapp{s}{\vec t}]\ {=}\
				            \elab{╬ö,\vec ¤ü}[T]
			            $$
			            where $T$ is defined using
			            $\mbf{chain}$ and $\mbf{pairs}$
			            thus:
			            $$
				            T Ôëö
				            \begin{scases}
					            \eapp{t_\cons}{\mbf{chain}(s, \vec t)}
					            &
					            \tsm{$\prn{╬▒ = \ch {t_\cons}}$}
					            \\
					            \eapp{t_\cons}{\mbf{pairs}(s, \vec t)}
					            &
					            \tsm{$\prn{╬▒ = \pw {t_\cons}}$}
					            \\
					            \eapp{t_\cons}{\vec t}
					            &
					            \tsm{$\prn{╬▒ = \al {t_\cons}}$}
				            \end{scases}
			            $$
			            $$
				            \begin{array}[t]{r@{\ {Ôëö}\ }l}
					            \mbf{chain}(s,\, t_1\ldots t_n)
					             & \eapp{s}{t_1\ t_2}\,
					            \eapp{s}{t_2\ t_3}
					            \,\ldots\,
					            \eapp{s}{t_{n-1}\ t_n}
					            \\[1mm]
					            \mbf{pairs}(s,\, t_1\ldots t_n)
					             & \eapp{s}{t_1\ t_2}
					            \,\ldots\,
					            \eapp{s}{t_1\ t_n}\,
					            \eapp{s}{t_2\ t_3}
					            \,\ldots\,
					            \eapp{s}{t_{n-1}\ t_n}
				            \end{array}
			            $$

		      \end{enumerate}

		      %

	\end{enumerate}

	%


\end{definition}

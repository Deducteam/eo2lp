\Autoref{fig:eo-syntax} defines syntax for the
fragment Eunoia considered in this document, where the
metavariable $s$ ranges over a set of \emph{symbols}.
%
First, we define grammars for the core term syntax.
%
Each term is either a symbol $s$, an (integer or string)
\emph{literal} $ℓ$, an \emph{application} $\eapp{s}{\vec t}$,
or a \emph{binder application} $\eapp{s}{\epar{\vec ρ}\ t}$.
%
Each \emph{parameter} $ρ$ consists of a symbol $s$, a term
% forward ref to where we discuss typing ??
$t$ (the type of the parameter), and possibly a
\emph{parameter attribute} $ν$.
%

Second, we define the \emph{commands} of Eunoia, which are
divided into \emph{standard commands} and \emph{proof commands}.
%
Hereinafter, a Eunoia \emph{signature} is a list $Δ$
of standard commands, and a \emph{proof script} is a
list $Υ = \prn{\vec δ ⨾ \vec π}$
where $\vec δ$ is a list of standard commands called
the \emph{preamble} and $\vec π$ is a list of
proof commands called the \emph{body} of the script.
%
In practice, a Eunoia-friendly SMT solver should provide a
signature $Δ$ that encodes the inference rules that may be
used in the proof scripts produced by the solver.
%
Then, the correctness of a proof script $Υ$ (wrt. $Δ$)
may be checked using the Ethos checker.
%

\subsection{Commands and their Declarations}\label{sec:eo-judge}
%
We define an abstract interface for reasoning about
Eunoia signatures and proof scripts.
%
This interface is useful for characterizing the
\emph{elaboration} and \emph{translation} operators
found in \autoref{sec:eo-elab} and \autoref{sec:results}
respectively.

\subsubsection{Constant Declaration.}
%
Let $δ$ be a \emph{(parameterized) constant declaration}
with symbol $s$, parameters $\vec ρ$, and term $t$.
%
We may write $\prn{⊢ s(\vec ρ) : t}$ to express
that the command $δ$ declares $t$ as the \emph{type} of $s$
with respect to parameters $\vec ρ$.
%
Furthermore, if a constant attribute $α$ is given by $δ$,
we may write $\prn{δ ⊢ s(\vec ρ) ∷ α}$.
%

\subsubsection{Macro Definition.}
%
Let $δ$ be a \emph{macro definition} with
symbol $s$, parameters $\vec ρ$, and term $t$.
%
Then $δ$ declares $t$ as the \emph{definiens} of $s$
wrt. $\vec ρ$; written $\prn{δ ⊢ s(\vec ρ) ≔ t}$.
%
If the attribute $\mtt{:type}\ t'$ is given by $δ$,
then $\prn{δ ⊢ s(\vec ρ) : t'}$ also holds.

\subsubsection{Inference Rule Declaration.}
%
Let $δ$ be a \emph{rule declaration} with symbol $s$,
parameters $\vec ρ$, and $\conclusion φ$.
\begin{enumerate}
	\item If $δ$ provides
	      $\requires{\ecase{l_1}{r_1}\ldots\ecase{l_k}{r_k}}$,
	      then let:
	      $$φ^★ ≔ \eoreqs{l_1}{r_1}{
			      \epar{
				      \,\ldots\,\eoreqs{l_k}{r_k}{φ}}
		      }
	      $$
	      Otherwise, $φ^★ ≔ φ$.

	      % TODO. "see ... for steps using rules with :premise-list".
	\item If $δ$ provides $\mtt{:premises}\ \epar{\plur ψ m}$,
	      then let:
	      $$τ^★ ≔ \earr{\eprf{ψ_1}\,\ldots\,\eprf{ψ_m}\ \eprf{φ^★}}$$
	      %
	      Otherwise, if $δ$ provides $\premiselist ψ f$ for
	      some formula $ψ$ and symbol $f$, then let:
	      $$τ^★ ≔ \earr{\eprf{ψ}\ \eprf{φ^★}}$$
	      Otherwise, let $τ^★ ≔ \eprf{φ^★}$.

	\item Suppose $δ$ provides $\mtt{:args}\ \epar{\plur t n}$
	      where each $t_i$ has type $τ_i$ with respect to the
	      current context $\prn{Δ, \vec ρ}$.
	      %
	      We introduce an \emph{auxilliary symbol}
	      $s^★$ with the following properties:
	      $$
		      \begin{array}[t]{l@{\ ⊢\ }l}
			      δ & s^★(\vec ρ) : \earr{\plur τ n\ \etype}
			      \\[1mm]
			      δ & s^★(\vec ρ) ≔ \mbf{cases}
			      [
				      \epar{{\eapp{s^★}{\plur t n}}\ τ^★}
			      ]
		      \end{array}
	      $$
	      Finally, the type of $s$ is given with an extended
	      list of parameters%
	      \footnote{
		      The symbols chosen for $\plur α n$ must be
		      \emph{fresh} with respect to $δ$.
		      That is, each $α_i$ is distinct from any symbol
		      occurring in any of the terms or parameters
		      supplied by $δ$.}
	      $\vec v$ thus:
	      $$δ ⊢
		      s(\vec v)
		      : \earr{\equote{α_1}\,\ldots\,\equote{α_n}\,\eapp{s^★}{\plur α n}}
	      $$
	      where $\vec v ≔
		      \vec ρ ⨾ \epar{α_1\ τ_1}\,\ldots\,\epar{α_n\ τ_n}$.
	      % TODO. what is actually the correct statement here?
	      The use of the auxilliary symbol ensures that the
	      `logical part' (i.e., the premises and conclusion)
	      of the rule may be obtained iff the given
	      arguments match the `patterns' specified by each $t_i$.
\end{enumerate}

\subsection{Elaboration of Terms}\label{sec:eo-elab}
%
% TODO.
% - need to define 'normal form' of Eunoia terms: the head
%   of every application must either be a builtin or a program.
Terms are \emph{elaborated} to eliminate all $n$-ary
applications and distinguish between constant-level and
program-level function applications.
%
We say that a term $t$ is in \emph{normal form} wrt. a
signature $Δ$ when the head symbol of every application in
$t$ is either a built-in or a $Δ$-program.

Consequently, an application $\eapp{f}{t}$ of an
unattributed constant symbol $f$ has the normal form
$\eapp{\mtt{\_}}{f\ t}$, where $\mtt{\_}$ is the Eunoia
built-in for higher-order application.
%
In general, the normal form of $\eapp{f}{\vec t}$
is determined by the attribute assigned to $f$ by $Δ$.
%
% %
% For example, consider a constant declaration $δ$ with:
% $$
% 	δ ⊢ \eor : \earr{\bool\ \bool\ \bool}
% 	\quad\text{and}\quad
% 	δ ⊢ \eor ∷ {\rcn \false}
% $$
% % and $δ ⊢ $.
% % $$\dca{\eor}{\earr{\bool\ \bool\ \bool}}$$
% %
% From the type of $\eor$, we may expect its only valid
% uses to be of the form $\eapp{\eor}{t_1\ t_2}$.
% %
% However, the assignment of the constant attribute means
% that the $\mtt{or}$ symbol is treated as
% \emph{right-associative} with \emph{nil-terminator} $\false$.
% %
% Thus, $n$-ary applications of $\eor$ are
% \emph{elaborated} to a normal form, e.g.;
% %
% $$\eapp{\eor}{\ev x\ \ev y\ \ev z}\ ⇢\
% 	\eapp{\eor}{\ev x\ \eapp{\eor}{\ev y\ \eapp{\eor}{\ev z\ \false}}}$$
% %
% Furthermore, the elaboration of such applications may
% also depend on the attributes of `locally bound' symbols.
% For example, consider $\vec ρ$ containing
% parameters $\epar{\ev x\ \bool}$,
% $\epar{\ev y\ \bool\ \mtt{:list}}$,
% and $\epar{\ev z\ \bool}$.
% %
% Then, observe:
% $$
% 	\eapp{\eor}{\ev x\ \ev y\ \ev z}
% 	\ {⇢}
% 	\ \econs\eor{\ev x}{
% 		\econcat{\eor}{\ev y}{\eapp{\eor}{\ev z\ \false}}
% 	}
% $$
% %
% Here, $\eo{list\_concat}$ is a built-in ternary operator:
% $\eapp{\eo{list\_concat}}{f\ \ell\ \ell'}$ concatenates
% two $f$-lists $\ell$ and $\ell'$.
% %
% The $\mtt{:list}$ attribute alters the elaboration strategy
% under the assumption that $\ev y$ will (eventually) be
% substituted for some $\eor$-list.
% %
% It may be helpful for the reader
% to consider the result of substituting
% $\ev y ↦ \econs\eor{\ev w}{\false}$
% thus:
% $$
% 	\begin{array}[t]{c}
% 		\econs\eor{\ev x}{
% 			\econcat{\eor}{\econs\eor{\ev w}{\false}}
% 			{\eapp{\eor}{\ev z\ \false}}}
% 		\\[1mm] {⇣} \\[1mm]
% 		\econs\eor{\ev x}{
% 			\econs\eor{\ev w}{
% 				\econs\eor{\ev z}{\false}}
% 		}
% 	\end{array}
% $$
% %
We define the $\mbf{elab}$ operator below with the aim
of supporting the elaboration strategies
corresponding to the constant attributes of
\autoref{fig:eo-syntax}.
%
\begin{definition}
	For any signature $Δ$, let $\elab{Δ,\vec ρ}$
	be the least (unary) operator such that:
	\begin{enumerate}
		\item For any symbol $s$, $\elab{Δ,\vec ρ}[s] = s$.
		      %
		\item Let $s$ be a symbol with $Δ ⊢ s ∷ \bd t_\cons$
		      (for some term $t_\cons$), and
		      let $\vec v = \epar{x_1\ t_1}\ldots\epar{x_n\ t_n}$.
		      Then for any term $t$, we have:
		      $$ \elab{Δ,\vec ρ}[\eapp{s}{\epar{\vec v}\ t}]
			      =
			      \eapp{s}{t_{\text{vars}}\ t_{\text{body}}}
		      $$
		      where
		      $t_{\text{vars}} ≔ \elab{Δ, \vec ρ}
			      [
				      \eapp{t_\cons}{
					      \eovar{\mtt{"}x_1\mtt{"}}{t_1}
					      \ldots
					      \eovar{\mtt{"}x_n\mtt{"}}{τ_n}
				      }
			      ]
		      $
		      encodes $\vec v$ as a list of $\eo{var}$ terms,
		      and $t_{\text{body}} ≔ \elab{Δ, \vec ρ}[t']$
		      where $t'$ is the result of replacing each
		      symbol $x_i$ in $t$ with its $\eo{var}$ encoding.
		      %
		\item For any symbol $s$ and list of terms $\vec t$:
		      $$
			      \begin{array}[t]{l}
				      \elab{Δ,\vec ρ}[\eapp{s}{\vec t}]\ {=}\
				      \\[2mm]
				      \quad
				      \begin{cases}
					      \foldr(G, t_\nil, \vec t')
					       & \tsm{if $Δ ⊢ s ∷ \rcn{t_\nil}$, }
					      \\
					      \foldl(G', t_\nil, \vec t')
					       & \tsm{if $Δ ⊢ s ∷ \lcn{t_\nil}$,}
					      \\
					      \foldr(G, t'_n, t'_1\ldots t'_{n-1})
					       &
					      \tsm{if $Δ ⊢ s ∷ \rc$,}
					      \\
					      \foldl(G', t'_1, t'_2\ldots t'_{n})
					       &
					      \tsm{if $Δ ⊢ s ∷ \lc$,}
					      \\
					      \elab{Δ,\vec ρ}[\eapp{{t_\cons}}{\mbf{chain}(s, \vec t')}]
					       &
					      \tsm{if $Δ ⊢ s ∷ \ch {t_\cons}$,}
					      \\
					      \elab{Δ,\vec ρ}[\eapp{{t_\cons}}{\mbf{pairs}(s, \vec t')}]
					       &
					      \tsm{if $Δ ⊢ s ∷ \pw {t_\cons}$,}
					      \\
					      \eapp{s}{{t_1'}\ldots{t_n'}}
					       &
					      \tsm{otherwise.}
				      \end{cases}
				      \\[3mm]
			      \end{array}
		      $$
		      where\footnote{%
			      $\foldl(G, a, l)$ (resp.\ $\foldr(G, a, l)$) is
			      the standard left fold (resp.\ right fold) using
			      combining function $G$ and accumulator $a$ on list $l$.
		      } $G ≔ \glue{\vec ρ, f}[x,y]$,
		      $G'(x,y) ≔ G(y,x)$,
		      and $t_i' ≔ \elab{Δ, \vec ρ}[t_i]$, and:

		      $$
			      \begin{array}[t]{r@{\ ≔\ }l}
				      % \glue{\vec ρ,f}[t_1,t_2] &
				      % \begin{cases}
				      %  \econcat{f}{t_1}{t_2} & \text{if $\vec ρ ⊢ t_1 ∷ \mtt{:list}$,}
				      %  \\
				      %  \econs{f}{t_1}{t_2}   & \text{otherwise.}
				      % \end{cases}
				      \mbf{chain}(f,\, t_1\ldots t_n)
				       & \eapp{f}{t_1\ t_2}\,
				      \eapp{f}{t_2\ t_3}
				      \,\ldots\,
				      \eapp{f}{t_{n-1}\ t_n}
				      \\[1mm]
				      \mbf{pairs}(f,\, t_1\ldots t_n)
				       & \eapp{f}{t_1\ t_2}
				      \,\ldots\,
				      \eapp{f}{t_1\ t_n}\,
				      \eapp{f}{t_2\ t_3}
				      \,\ldots\,
				      \eapp{f}{t_{n-1}\ t_n}
			      \end{array}
		      $$
	\end{enumerate}

	%


\end{definition}

\subsection{Built-in Operators}\label{sec:eo-builtin}
%
% ---------------------------------------------------------
\begin{boxfigure}[t!]{fig:eo-builtins}
	{A selection of built-in operators of Eunoia.}
	%
	$$ \eoBuiltinFig $$
\end{boxfigure}
% ---------------------------------------------------------
%
In addition to the user-defined symbols introduced by
commands, Eunoia provides a fixed set of \emph{built-in
	operators} (\autoref{fig:eo-builtins}).
%
Built-in operators are evaluated eagerly on ground
arguments, so that e.g.\ $\eapp{\eo{add}}{1\ 1}$ is
syntactically equal to $2$ in all contexts.
%
Apart from $\eo{ite}$, evaluation proceeds bottom-up:
arguments are fully reduced before the operator is applied.
For $\eo{ite}$, only the condition is evaluated; branches
are left unevaluated until the condition resolves.
%
We briefly describe the main categories below.

\subsubsection{Core Operators.}
%
The operator $\eo{eq}$ tests syntactic equality of two
ground values, returning $\mtt{true}$ or $\mtt{false}$.
%
$\eo{ite}$ is a conditional: $\eapp{\eo{ite}}{b\ t_1\ t_2}$
returns $t_1$ if $b$ is $\mtt{true}$ and $t_2$ if $b$ is
$\mtt{false}$.
%
$\eo{requires}$ acts as a guard:
$\eapp{\eo{requires}}{t_1\ t_2\ t_3}$ returns $t_3$ when
$t_1$ and $t_2$ are syntactically equal, and is unevaluated
otherwise.
%
This operator is central to rule declarations
(cf.~\autoref{sec:eo-judge}), where it constrains the
applicability of inference rules to contexts satisfying
certain equalities.
%
The operator $\eo{typeof}$ returns the type of a ground
value, and $\eo{var}$ constructs a variable from a string
name and a ground type.

\subsubsection{Boolean and Arithmetic Operators.}
%
The Boolean operators $\eo{and}$, $\eo{or}$, $\eo{not}$,
and $\eo{xor}$ have the expected semantics on $\mtt{true}$
and $\mtt{false}$.
%
Arithmetic operators include $\eo{add}$, $\eo{mul}$,
$\eo{neg}$, $\eo{is\_neg}$, integer division $\eo{zdiv}$
and modulus $\eo{zmod}$.
%
These operate on integer (and rational) literals; for
example, $\eapp{\eo{add}}{3\ 4}$ evaluates to $7$
and $\eapp{\eo{is\_neg}}{-2}$ evaluates to $\mtt{true}$.

\subsubsection{List Operators.}
%
A key feature of Eunoia is its treatment of $n$-ary
operators via \emph{$f$-lists}.
%
Given a right-associative operator $f$ with nil terminator
$t_\nil$, an $f$-list with elements $\plur t n$ is the
term $\eapp{f}{t_1\ \eapp{f}{\ldots\ \eapp{f}{t_n\ t_\nil}}}$.
%
The built-in list operators manipulate these structures
generically.
%
$\eapp{\eo{nil}}{f\ T}$ returns the nil terminator of $f$
specialized to type $T$.
%
$\eapp{\eo{cons}}{f\ t_1\ t_2}$ prepends $t_1$ to the
$f$-list $t_2$.
%
$\eapp{\eo{list\_concat}}{f\ t_1\ t_2}$ concatenates two
$f$-lists.
%
Further operators include $\eo{list\_len}$,
$\eo{list\_nth}$, $\eo{list\_find}$ (length, indexing,
and search), as well as $\eo{list\_rev}$ (reversal),
$\eo{list\_erase}$ (removal), and multiset operations
such as $\eo{list\_minclude}$ and $\eo{list\_meq}$.
%
These operators are used extensively in the CPC signature
to define programs that normalize and compare $n$-ary
terms.

%
In addition to $f$-lists, Eunoia provides a
\emph{heterogeneous} list type $\eo{List}$ with
constructors $\eo{List::cons}$ and $\eo{List::nil}$.
%
These are used to encode binder variable lists and argument
lists that may contain elements of different types.
